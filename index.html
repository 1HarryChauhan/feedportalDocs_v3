Getting Started Overview

What You'll Learn: Once you're ready to start contributing to this project, you'll need to use tools like GitHub. Think of GitHub as a way to keep a safe copy of the entire project online. You can ask for a copy anytime, work on it, complete some extra features or fixes, and then submit your work for checking (called "merging"). This way, everyone can work together without breaking each other's code!

Prerequisites (accounts, tools, access requests)

What You Need Before Starting: Before you can start working on the project, you'll need access to several tools and accounts. This section covers everything you need to request or set up.

Before getting started, ensure you have access to:

  • GitHub Account: Create a free account at github.com and share your username with the team leads
  • Microsoft Teams Access: Request access to the project's Teams workspace from your instructor
  • OnTrack Account: Your university login should provide access to OnTrack for task management
  • Development Tools: We'll install these in the next steps (Node.js, Git, MySQL, code editor)

Complete this checklist before starting development:

  • □ OnTrack account accessible and project visible

1. Clone the Repository

What is Cloning? Cloning means downloading a complete copy of the project to your computer. It's like making a photocopy of a book - you get everything: all the code files, folders, and history. This way you can work on your own copy without affecting the original.

Start by cloning the project repository to your local machine:

git clone <repository-url> cd FeedFormulationPortalV2
Primary Repository

2. Create Your Feature Branch

What is a Branch? Think of a branch like creating a separate workspace for your changes. It's like making a copy of a document to edit while keeping the original safe. Everyone works on their own branch so they don't interfere with each other's work. When you're done, you can merge your changes back to the main project.

Before starting work, create a new branch using the following naming convention:

git checkout -b username/feature-name

Example:

git checkout -b kush/update-feed-api

Branch Naming Convention:

  • Use your username followed by a forward slash.
  • Use hyphens to separate words in the feature name (not underscores).
  • Keep feature names descriptive but concise.
  • Use lowercase letters.
  • Include the type of change (e.g., api, fix, feat, docs).
  • Examples: john/feat-add-user-auth, sarah/api-update-feed, mike/fix-db-connection.

3. Environment Setup

What are Environment Files? Environment files (.env) are like secret configuration files that contain passwords, database addresses, and other sensitive information. They're kept separate from the main code for security reasons. Think of them as a settings file that tells your application how to connect to databases and other services.

Find env files from the project's Files tab in Microsoft Teams, then rename both files to .env and place them in their respective directories (backend/ and frontend/).

Update the database credentials in backend/.env:

DB_HOST=localhost DB_USER=your_username DB_PASSWORD=your_password DB_NAME=feed_portal

4. Database Setup

What is a Database? A database is like a digital filing cabinet where all the application's information is stored - user accounts, feed recipes, ingredients, etc. MySQL is the type of database we're using. Setting it up means creating the "filing cabinet" and organizing it with the right folders (tables) to store our data properly.

Create a new MySQL database:

CREATE DATABASE feed_portal;

Set up the tables using the SQL files in backend/sql/:

  • Run ingredients.sql to create the ingredients table.
  • Run feeds.sql to create the feeds and association tables.

Use MySQL command line or a tool like MySQL Workbench:

mysql -u your_username -p feed_portal < backend/sql/ingredients.sql mysql -u your_username -p feed_portal < backend/sql/feed.sql

5. Install Dependencies

What are Dependencies? Dependencies are like ingredients in a recipe - they're pre-built pieces of code that our application needs to work properly. Instead of writing everything from scratch, we use these ready-made tools (libraries) that other programmers have created. npm install downloads all these "ingredients" for us.

Install the required dependencies for both backend and frontend:

Backend:

cd backend npm install

Frontend:

cd frontend npm install

6. Start the Application

Frontend vs Backend: Think of the frontend as the "face" of the application - what users see and click on (like buttons, forms, and pages). The backend is like the "brain" - it handles data, processes requests, and talks to the database. Both need to run together for the application to work completely.

Start the backend and frontend development servers:

Backend:

cd backend npm run dev

Frontend:

cd frontend npm start

The application will be available at:

  • Frontend: http://localhost:3000
  • Backend API: http://localhost:5000

Common Errors and Development Tips

Essential Reading: This guide contains critical information that will save you hours of debugging and prevent common mistakes that have caused major issues in the past.

Critical Database Rules

Rule #1: Master Database Control

Never let anything through unless it was checked by the one person who has understanding of the project direction.

We have had massive arguments over merges that would have screwed the user table which Clerk (our entire authentication system) is hooked into. It took two trimesters to get it working right.

Why This Matters: Database modifications can break authentication, corrupt user data, and require extensive manual repairs. Always get database changes reviewed by a lead developer before merging.

Sprint and Task Management

Plan for Incomplete Work

  • Leave tasks for the next sprint and trimester - it makes everything much easier
  • Do not expect everyone to finish their work
  • Assign work to people who can do it if it is of great importance
  • Otherwise you will be doing it yourself and the other person will pass on your work

Git and Merging Best Practices

Never Merge Your Own Work

Do not merge your work - let a lead test it in their testing threads and let them merge.

Many errors and nightmares we have had over people just merging work. Often they break things, we miss it for a few weeks, and then we have to play find the needle in the haystack and hand code a lot of the project again by hand.

Proper Branch Management

Don't Follow School Suggestions Exactly: The school suggests making branches and updating them at the start of the sprint, then submitting them in PR when ready. Don't do it exactly like this or you will end up with your leads upset due to you being massively behind the main repo.

Correct Branching Workflow

  1. Make the branch at the start of the sprint or task work on it
  2. When you have finished your work, make a new branch with -rebase at the end of its name
  3. Put all of your work into the new code base so you have all the new additions to the code base in your branch and you're up to date with the main branch's commits
  4. This can be done in the browser or on your machine with a text editor
Example branch naming: Original: feature/user-auth Rebase branch: feature/user-auth-rebase

Debugging and Troubleshooting

Browser Caching Issues

Common Problem: If you have been working on your project and you update it but do not see updates, remember your browser cookies and refresh the browser so that it actually downloads the new code rather than caching old code.

Clear Browser Cache Steps:

  1. Hard refresh: Ctrl + F5 (Windows) or Cmd + Shift + R (Mac)
  2. Clear browser cache and cookies
  3. Open developer tools and disable cache while DevTools is open
  4. Use incognito/private browsing mode for testing

Server-Side Debugging

When running your code, if you come to troubles, make sure you check the terminals for each of the servers. There are often many codes that will give you hints on how to debug your code errors given from the servers through their terminals.

Console Logging for Debugging

// Use console.log() for debugging console.log('Input arguments:', arg1, arg2); console.log('Variable value:', someVariable); console.log('Function result:', result);

API Testing with Postman

Setting Up Postman Authentication

When testing endpoints, use Postman. You will need to get a Clerk authentication token to do this.

Steps to get authentication token: 1. Open the project in your browser (localhost:3000) 2. Open the development console 3. Enter this into the console and run it: await window.Clerk.session.getToken() 4. Copy the key displayed in the output log pane 5. In Postman, go to Authorization tab 6. Select "Bearer Token" as the token type 7. Paste your copied token

Emergency Procedures

If You Break the Main Branch

  1. Immediately notify team leads via Teams
  2. Do not attempt to fix it yourself
  3. Provide details about what changes were made
  4. Be prepared to help with rollback procedures

Project Overview

The Feed Formulation Portal is a web-based application designed to assist nutritionists, farmers, and feed manufacturers in creating optimized animal feed formulations. Developed as a Deakin University Capstone project by Gopher Industries, it leverages modern web technologies to provide an intuitive interface for managing ingredients, calculating nutritional requirements, and generating compliant feed recipes.

The portal addresses key challenges in the animal nutrition industry, such as ensuring regulatory compliance, minimizing costs, and maximizing nutritional balance. It supports various livestock types (e.g., poultry, cattle, swine) and integrates real-time calculations for nutrient optimization.

Key Features

  • Ingredient Management: Database of ingredients with nutritional profiles, costs, and availability tracking.
  • Formulation Engine: Linear programming-based optimizer for least-cost formulations meeting nutritional constraints.
  • User Authentication: Secure login via Clerk, with role-based access (admin, user, guest).
  • Recipe Generation: Export formulations as PDFs or CSV files with compliance reports.
  • Analytics Dashboard: Visualize formulation history, cost trends, and performance metrics.

System Architecture

The architecture follows a client-server model with RESTful APIs for communication. The frontend handles UI/UX, while the backend manages business logic and data persistence.

Use Cases

  • Farmers formulating daily rations for herds.
  • Nutritionists testing hypothetical scenarios for research.
  • Manufacturers scaling production while adhering to regulations like NRC standards.

Network & Connectivity Issues

Symptom: API calls failing with CORS errors.

Solution: Ensure backend runs on port 5000 and frontend on 3000. Add CORS middleware in Express if needed.

Authentication Problems

Symptom: "Invalid token" errors.

Solution: Regenerate Clerk token via browser console. Verify .env CLERK keys match dashboard.

Database Errors

Symptom: Connection refused.

Solution: Check MySQL service status. Verify DB_HOST, USER, PASS in .env. Restart server if schema mismatch.

Deployment Troubleshooting

  • Environment variables not loading: Use process.env checks with fallbacks.
  • Build failures: Run npm audit fix for vulnerabilities.
  • Performance issues: Profile with Node clinic or React DevTools.

Branching Strategy

  • Main: Stable production branch.
  • Develop: Integration branch for features.
  • Feature branches: username/feat-description.
  • Rebase before PR: git rebase develop.

Commit Conventions

feat: add user login
fix: resolve db query bug
docs: update readme

Pull Requests

  1. Create PR from feature to develop.
  2. Include description, screenshots, tests.
  3. Assign reviewer (lead).
  4. Merge only after approval.

Code Reviews

Focus on: Security, performance, readability. Use checklists in PR templates.
